Технічна шпаргалка для проєкту. Без води, тільки факти.

## 2. Структура БД (SQLite)

```
Users              (Id, Username, PasswordHash, Email, Role, CreatedAt)
Artists            (Id, Name, Aliases, Country, Description)
                   -- Aliases: TEXT NULL, через кому ("Death Grips, DG, MC Ride") — для нечіткого пошуку
Genres             (Id, Name)
Albums             (Id, ArtistId, Title, Year, GenreId, CoverPath, Description)
Tracks             (Id, AlbumId, Position, Title, Lyrics, Duration, SamplePath, FullPath)
                   -- Lyrics: TEXT NULL, для повнотекстового пошуку по словах пісень
Products           (Id, AlbumId, Format, Price, Stock, ReleaseYear, Label, IsActive)
                   -- Format: enum {Vinyl, CD}
CartItems          (Id, UserId, ProductId, Quantity, AddedAt)
Orders             (Id, UserId, CreatedAt, Status, TotalAmount)
                   -- Status: enum {New, Processing, Completed, Cancelled}
OrderItems         (Id, OrderId, ProductId, Quantity, UnitPrice)
Reviews            (Id, ProductId, UserId, Text, Rating, CreatedAt)
                   -- Rating: 1..5
Playlists          (Id, UserId, Name, CreatedAt)    -- існуюча фіча плеєра, переноситься з M3U в БД
PlaylistTracks     (PlaylistId, TrackId, Position)
PlayerSettings     (UserId, Volume, RepeatMode, ShuffleMode, LastTrackId)

-- Пошукова підсистема (АІПС-ядро):
SearchIndex        FTS5 virtual table: (content_type, content_id, title, body)
                   -- content_type: 'artist' | 'album' | 'track' | 'review'
                   -- tokenize: 'unicode61 remove_diacritics 2'
                   -- забезпечує BM25 ранжування з коробки
SearchHistory      (Id, UserId, Query, ExecutedAt, ResultCount)
SavedSearches      (Id, UserId, Name, QueryJson, NotifyOnNew, CreatedAt)
```

**Ключові зв'язки:**
- `Album` 1—N `Track`
- `Album` 1—N `Product` (один альбом може мати дві версії: вініл і CD)
- `Order` 1—N `OrderItem`
- `Product` 1—N `Review`
- `User` 1—N `Order`, `Review`, `Playlist`, `CartItem`
- Доступ до повного відтворення: `JOIN Orders → OrderItems → Products → Album → Tracks` фільтруючи по `UserId` і `Status = Completed`.

## 5. Міні-плеєр унизу — поведінка

**Стан 1 — прихований (за замовчуванням при запуску):** нижня панель відсутня, основна область займає всю висоту.

**Стан 2 — поява:** при `IPlayerService.MediaOpened` (тобто щось почало грати — чи 30-сек семпл, чи повний трек) панель **плавно з'являється знизу**:
- `TranslateTransform.Y`: 72 → 0
- `Opacity`: 0 → 1
- Тривалість: 250мс, easing `CubicEaseOut`

**Стан 3 — користувач закрив [✕]:** дзеркальна анімація вниз, панель ховається. Музика **продовжує грати** (просто без видимих контролів). При наступному `MediaOpened` (новий трек / новий семпл) — з'являється знов.

**Кнопка [⛶] (розгорнути):** перемикає бічну навігацію на вкладку «Плеєр», тобто відкриває повний UI плеєра. Міні-плеєр сам ховається.

**Контент міні-плеєра:**
- мала обкладинка 56×56,
- назва треку (`TextBlock` обрізається по `TextTrimming="CharacterEllipsis"`),
- виконавець (менший шрифт),
- кнопки Prev / Play-Pause / Next,
- progress bar (не drag-абельний для простоти — тільки індикатор),
- регулятор гучності,
- кнопки [⛶] і [✕].

## 8. Пошукова підсистема (ядро АІПС)

Саме цей розділ відрізняє «магазин з фільтрами» від справжньої інформаційно-пошукової системи. Тут описано, що працює під капотом єдиного поля пошуку з 4.1.

### 8.1. Індексація — SQLite FTS5

Окрема virtual table `SearchIndex` зі стрічками `content_type`, `content_id`, `title`, `body`. Перевага FTS5 — **вбудоване BM25 ранжування** (класична модель з теорії інформаційного пошуку). Tokenizer: `unicode61 remove_diacritics 2` — нормалізує реєстр і діакритику, прибирає різницю між «Бьорк» і «Бьоrk».

Індексуються:
- виконавці: `Name + Aliases`;
- альбоми: `Title + Description`;
- треки: `Title + Lyrics`;
- тексти відгуків.

Тригери на `INSERT/UPDATE/DELETE` основних таблиць оновлюють індекс автоматично.

### 8.2. Парсер запитів

Власний парсер (рекурсивний спуск) розпізнає:
- **вільний текст**: `death grips guillotine`;
- **польові оператори**: `виконавець:"death grips"`, `жанр:jazz`, `формат:lp`;
- **діапазони**: `рік:2010..2015`, `ціна:<600`, `рік:>1970`;
- **виключення**: `-блюз`;
- **точну фразу**: `"the money store"`.

Парсер будує AST → транслюється в комбінований запит (FTS5 `MATCH` для тексту + звичайний `WHERE` для атрибутів).

### 8.3. Фасетна навігація

Класичний IR-патерн: після основного запиту виконуються `GROUP BY` запити по кожному фасету (жанр, рік, формат, ціна, рейтинг) із підрахунком кардинальностей. Лічильники показуються поряд із чекбоксами і **перераховуються динамічно** при зміні будь-якого фільтра.

### 8.4. Автодоповнення

Префіксний пошук `SearchIndex MATCH 'deat*'`, відсортовано за **популярністю**:
```
popularity_score = sales_count × 0.7 + log(1 + view_count) × 0.3
```
Debounce 200мс на боці UI, щоб не довбати БД на кожне натискання клавіші.

### 8.5. Нечіткий пошук («Чи мали ви на увазі»)

Якщо точних результатів < 3:
- триграмний пошук поверх кандидатів;
- ранжування за подібністю Jaccard (або відстань Левенштейна < 3 для коротких запитів);
- найкращий варіант пропонується посиланням «Чи мали ви на увазі: …».

### 8.6. Історія та збережені запити

- `SearchHistory` пише кожен виконаний запит (для аналітики, автодоповнення на порожньому стані, статистики «що шукають»).
- `SavedSearches` дає користувачу зберегти набір фільтрів з прапорцем `NotifyOnNew`: коли з'являється товар, що матчить умови, у профілі з'являється індикатор.

### 8.7. Модель ранжування

Фінальна оцінка для кожного результату:
```
score = BM25(query, document)  × 0.6
      + log(1 + sales_count)   × 0.2
      + (rating / 5.0)         × 0.1
      + recency_factor         × 0.1
```
де `recency_factor = exp(-Δdays / 365)`. Ваги — параметри підсистеми, підбираються емпірично; на захисті обґрунтовуються як **гіперпараметри моделі ранжування**.

## 9. Чого свідомо НЕ робимо

- Не пишемо власну систему оплати — статус замовлення міняється адміном вручну.
- Не пишемо рекомендаційний алгоритм («подібні альбоми») — поза scope.
- Не робимо мобільну версію — Avalonia це вміє, але теза «настільний застосунок» обмежує scope.
- Не пишемо повний `MusicBrainz` lookup для метаданих — TagLibSharp читає теги з самих файлів, цього достатньо.
- Не робимо багатокористувацький онлайн (магазин не синхронізується між машинами) — БД локальна, кожна копія самостійна. Для дипломної цього достатньо.
